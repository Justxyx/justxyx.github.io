# 本地精确测量 nginx + Linux 协议栈一次会话耗时

原生耗时  +  简单插件耗时 + 复杂正则匹配耗时 等等， 可以整理表格看看

评审主席团会有疑问， 为什么不直接在本地写for循环记录时间， 再对比nginx 的整体时间， 这样就可以看加了循环的代码对整体性能的拖累

单独写 for 循环测时间，只能得到“代码本身的理论成本”，
而 nginx 整体性能受调度、缓存、流水线、并发模型影响，
只有在真实 nginx 请求路径中测量，才能得到“真实可感知的性能拖累”。

**如果只看 for 循环时间就够了，那我们为什么还要做线上压测？**

## 1. 目标

在**本地 loopback（lo）接口**上，精确测量：

* 一次完整请求-响应（RTT）耗时
* nginx 自身处理耗时
* 新增插件 / 新增代码（甚至一个 for 循环）带来的额外时间

用于回答以下工程问题：

* nginx + 协议栈完成一次会话要多久？
* 插件到底慢了多少？
* 多写一段代码是否“值得”？

---

## 2. 为什么必须用 loopback（lo）

Linux 的 `lo` 接口特点：

* 无物理网卡
* 无中断、无 DMA
* 无驱动队列
* 不经过真实链路层

因此：

> 在 lo 上测到的时间，几乎完全是“软件执行时间”

非常适合做 **性能对比实验**。

---

## 3. 一次请求的完整时间路径

```
Client(user)
  ↓
Linux TCP/IP 协议栈
  ↓
nginx (accept → read → process → write)
  ↓
Linux TCP/IP 协议栈
  ↓
Client(user)
```

测量目标可拆解为：

* 整体 RTT（用户视角）
* nginx 内部处理时间
* 插件 / 新增逻辑的增量时间

---

## 4. 基础测试方案（推荐）

### 4.1 测试拓扑

```
wrk / 自定义 client
   ↓ lo
nginx（原始）
nginx（+ 插件 / + 新代码）
```

### 4.2 测试原则

* 同一台机器
* 同一 nginx 配置
* 同一并发 / 请求模型
* 一次只改一处代码

---

## 5. nginx 内部精确计时（核心）

### 5.1 推荐时钟

**必须使用：**

```c
clock_gettime(CLOCK_MONOTONIC_RAW, &ts);
```

**禁止使用：**

* gettimeofday
* CLOCK_REALTIME

原因：避免 NTP 校时、时间回拨、频率修正干扰。

---

### 5.2 nginx 内部推荐打点位置

```
T1: accept 完成
T2: 请求读完
T3: handler 开始
T4: handler 结束
T5: 响应写完
```

可得到：

* nginx 纯处理时间：`T4 - T3`
* nginx 总耗时：`T5 - T1`
* 插件引入的额外耗时：对比前后差值

---

## 6. 客户端侧：测整体 RTT

客户端只测一件事：

```
send(request) → recv(response)
```

记录：

```
RTT = t_end - t_start
```

这是**最终用户视角的真实延迟**。

---

## 7. for 循环级别的精确测量（微基准）

### 7.1 为什么 RTT 不够用

* RTT 噪声：µs 级
* for 循环成本：ns～µs 级

结论：

> 微小代码开销必须用微基准测试

---

### 7.2 正确做法：CPU cycle 级测量

```c
static inline uint64_t rdtsc(void)
{
    uint32_t lo, hi;
    __asm__ volatile ("rdtsc" : "=a"(lo), "=d"(hi));
    return ((uint64_t)hi << 32) | lo;
}
```

示例：

```c
uint64_t t1 = rdtsc();

for (int i = 0; i < N; i++) {
    x += i;
}

uint64_t t2 = rdtsc();
```

计算：

```
cycles = t2 - t1
time(ns) = cycles / CPU_freq
```

| ------------- | -------- |

| nginx handler | µs 级     |

| for 循环        | ns～µs 级  |

## 9. 测试环境要求（否则数据不可信）

必须满足：

* CPU governor = performance
* 关闭 turbo boost
* nginx / client 绑核（taskset）
* 尽量单核测试

## 10. 最终结论

> 在本地 loopback（lo）环境下，通过：
>
> * nginx 内部精确打点
> * 客户端 RTT 对比
> * 微基准（rdtsc）测量
>
> 可以非常严谨、可复现地评估：
>
> * 插件是否拖慢性能
> * 拖慢了多少
> * 是否值得上线

这是**高性能网络服务的标准工程方法**。
